§ 02

Before you start

Host prerequisites

The MCP server runs on some machine — your laptop or a workstation. That host needs:

ToolNeeded forNotes
Python 3.10+ · mcp · requeststhe server + every Supabase/Vercel API callpip3 install -r requirements-optional.txt
vercel CLIthe Vercel deploy stepnpm i -g vercel. Builds in Vercel's cloud — no local node_modules

node / npmonly to run the site locallynot required to deploy
Pure REST

Provisioning Supabase and deploying the edge function need no Supabase CLI, Deno, or Docker — they go straight over the Management API.

Register the MCP server

Add the server to your agent (Claude Desktop / Cursor / Claude Code) and put the tokens in the server's environment so they never appear in chat:

claude_desktop_config.json
{
  "mcpServers": {
    "corrupt-engine": {
      "command": "python3",
      "args": ["/abs/path/to/corrupt-cli/corrupt-mcp/server.py"],
      "env": {
        "SUPABASE_ACCESS_TOKEN": "sbp_...",
        "VERCEL_TOKEN": "..."
      }
    }
  }
}

Every tool also accepts tokens as arguments, but the env block is the lower-exposure option.

Get your tokens

  • Supabase personal access token (SUPABASE_ACCESS_TOKEN) — from Dashboard → Account → Tokens. It must start with sbp_. A project key (sb_secret_…) will not work for the Management API. It can touch every project in your orgs — treat it like a root credential.
  • Vercel token (VERCEL_TOKEN) — from Vercel → Account → Tokens. Scope it to your team if you use Vercel Teams.
  • Resend API key (optional, resend_api_key). Without it the site works — lead emails are just skipped until you set it; the lead still saves and shows in the admin inbox.
Record

The generated database password and webhook secret come back in the tool output. Save them — Supabase can't show the DB password again.

§ 03

Collect the client brief

Have these ready before you run anything.

Required

  • Brand name (e.g. Acme Motors)
  • Phone — any format; digits are extracted for tel: links
  • Public contact email
  • Street, city, state, ZIP
  • Bare domain, no protocol (e.g. example.com)
  • Super-admin email — the first person who can log into the admin portal. Until this row exists, nobody can (RLS is the whole access layer).

Optional

  • legal_suffix (default LLC), legal_name, tagline, founded_year, facebook_url
  • locations_json — a JSON array of lots (defaults to one main lot). IDs are lowercase / hyphen / underscore, no spaces:
    locations_json
    [{"id":"main","label":"Main Lot"},{"id":"north","label":"North Lot"}]
  • Email delivery: resend_api_key, notification_to_emails (comma-separated), notification_sender_email (a verified Resend address), notification_brand_name
  • Infra: region (default us-east-1), organization_slug (only if your account has more than one org), Vercel team_id

Moment of truth

§ 05

Step by step

Run the tools individually to inspect between stages, retry one step, or split provisioning from deployment. Order matters — each feeds the next.

scaffold_dealership_pro

Copies the template to target_path (outside the repo; excludes node_modules/.next/.git/env files) and generates src/config/site.ts from the brief. No infrastructure touched. force: true overwrites a non-empty target. → returns target_path, site_config, env_local.

provision_supabase_dealership_pro

The heavy step. Creates the project (or reuses existing_ref), waits until healthy, then over the Management API: fetches the URL + browser key, applies the schema (skipped if already applied), seeds the super admin, sets the three Vault secrets, and sets the edge-function secrets. → returns ref, project_url, anon_key, db_password, supabase_db_url (a Session Pooler URI for psql), webhook_secret, and dashboard.

One secret, two homes

The same webhook secret is used for both the Vault secret the DB trigger sends and the function's WEBHOOK_SECRET — kept in sync automatically so notifications can't fail on a mismatch.

deploy_function_dealership_pro

Deploys send-credit-app-notification via the Management API (Supabase bundles it server-side). Needs ref. verify_jwt defaults to False on purpose — see §10. The database “webhook” is the AFTER INSERT trigger the schema already built, so nothing else needs wiring. → returns the function endpoint.

deploy_vercel_dealership_pro

Creates (or reuses) the Vercel project, wires NEXT_PUBLIC_SUPABASE_URL + NEXT_PUBLIC_SUPABASE_ANON_KEY, writes .vercel/project.json, and deploys with the vercel CLI (cloud build). Needs target_path and the Supabase URL + key. → returns the live url.

verify_dealership_pro

Runs rls_regression.sh as the security gate. Seeds two throwaway @…invalid admin identities, runs the suite, then removes only the rows it inserted. Give it ref + target_path, and either supabase_db_url or db_password. Needs psql + bash. A clean run is 27/27.

§ 06

After the tools — manual steps

The automation gets you a working site at *.vercel.app. These last-mile items are deliberately manual — they touch registrars, third-party verification, or human logins.

Point the custom domain

  1. Vercel → project → Settings → Domains: add www.<domain> and the apex.
  2. At the registrar, add the records Vercel shows (usually a CNAME to cname.vercel-dns.com).
  3. On Cloudflare, set the records to “DNS only” (grey cloud) — Vercel's Let's Encrypt SSL fails behind the orange-cloud proxy. Confirm routing with dig @8.8.8.8 <domain>, not just Vercel's checkmarks.

Verify the Resend sending domain

Resend won't send from an unverified domain. Verify the client's domain and set notification_sender_email to an address on it — or, to test, use Resend's sandbox sender onboarding@resend.dev to your own inbox. Skipped Resend entirely? Leads still save and appear in the admin inbox; set the key later to turn on email.

First admin login & inventory

Send the client to <site>/admin. They enter the seeded super-admin email, get an OTP by email, and they're in. Inventory is entered by hand in the Inventory tab — add vehicles, upload photos. There is no DMS sync by design.

§ 07

The security gate

rls_regression.sh asserts what each policy is supposed to allow, for every identity that matters — anonymous visitor, self-registered stranger, ordinary admin, super admin — plus the storage rules and the structural invariants (exactly three wide-open policies; RLS on every table; no TO public all-command policy). It simulates identities in rolled-back transactions and writes nothing, so it's safe against production. Run it after any later change under supabase/. Green is 27 assertions passing.

§ 08

Re-runs, idempotency & recovery

  • Provisioning is idempotent. The schema applies at most once (gated by the authorized_admins marker). Re-running with existing_ref re-seeds the admin and re-sets secrets without touching the schema, and reuses the existing webhook secret instead of rotating it.
  • A failure after the project is created preserves the ref. The error names it and tells you to re-run with existing_ref=<ref> — so you never orphan a paid project or make a duplicate.
  • No force_reapply. The baseline is a one-shot “apply to an empty project” script, not idempotent DDL. To start clean, delete the project and create a fresh one.
  • scaffold won't overwrite a non-empty target without force: true, and it refuses any target_path inside the repo.

§ 10

What's not automated — accepted risks

  • RLS is the entire access-control layer. The browser uses only the anon / publishable key; there is no server API. Never put the service_role key in the app.
  • verify_jwt=False on the edge function is deliberate. The only caller is the DB trigger, which sends the project's browser key; on projects created since Nov 2025 that's a publishable key (not a JWT), so a gateway JWT check would 401 the trigger's fire-and-forget call and silently kill lead emails. The function authenticates itself with a fail-closed X-Webhook-Secret check.
  • No rate limiting on the public lead/error inserts — honeypots only. Both forms are public by design.
  • is_email_authorized is anon-callable (an admin-email oracle) — required, since it runs before login; knowing an address is an admin grants nothing without that inbox.
§ 11

Teardown

To retire a stand-up, the operator deletes the pieces — these are destructive and are your action, not the tool's:

  • Supabase — Dashboard → project → Settings → General → Delete project.
  • Vercel — project → Settings → Delete Project.
  • DNS — remove the records you added at the registrar.
  • The generated folder under target_path is just files — delete it.
